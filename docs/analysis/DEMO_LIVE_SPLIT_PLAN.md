# Demo/Live Split Plan

## Нужно ли разделять demo и live?

Да. На текущем этапе это одно из самых полезных улучшений архитектуры.

## Почему

- Сейчас часть demo/live ветвлений находится в UI, часть — в сервисах.
- Это усложняет сопровождение и может приводить к рассинхрону поведения.
- Разделение режимов уменьшает количество условной логики в компонентах.

## Что уже сделано в этом шаге

- Добавлена единая точка определения режима: `src/config/runtimeMode.js`.
- `firebaseService` переведён на `isLiveMode` из runtime-конфига.
- `ChatView` и `useCreateRequest` частично отвязаны от строковых проверок режима.

## Следующие шаги

1. Вынести все demo/live проверки из компонентов в domain/services слой.
2. Добавить интерфейсы-адаптеры `RequestsGateway`, `ChatGateway`, `UsersGateway`.
3. Реализовать два провайдера: `demoGateway` и `firebaseGateway`.
4. Подключать провайдер через единый контейнер/фабрику на старте приложения.
5. Закрыть модульными тестами совпадение контрактов между gateway-реализациями.

## Критерий готовности

- UI-компоненты не содержат проверок режима.
- Переключение demo/live не требует правок UI-кода.
- Все side effects идут через единый gateway-контракт.
