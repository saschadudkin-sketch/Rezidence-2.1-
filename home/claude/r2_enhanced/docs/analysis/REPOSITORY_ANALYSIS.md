# Repository Analysis — Rezidence-2

Краткий структурированный обзор приложения для системы управления доступом «Резиденции Замоскворечья».

## Что это за приложение

- React SPA (Create React App) для контроля доступа в жилом комплексе.
- Основные сценарии: гостевые/рабочие пропуска, техзаявки, пост охраны, чат, чёрный список, админ-панель.
- По умолчанию работает в demo-режиме с локальными данными и localStorage; предусмотрены точки расширения для Firebase live-режима.

## Технологии

- React 18, react-scripts 5 (CRA).
- Firebase SDK как зависимость, но runtime по умолчанию отключён фича-флагом.
- QR-коды через пакет `qrcode`.

## Архитектура

- Вход: `src/index.js` → `src/App.jsx`.
- Глобальный стейт: `src/store/AppStore.jsx` с `useReducer` и набором слайсов.
- Модуль прав доступа: `src/domain/permissions.js`.
- UI по ролям: `src/views/*`.
- Инфраструктурные адаптеры: `src/services/firebaseService.js`, `src/services/logger.js`.

## Роли и доступ

- Роли: owner, tenant, contractor, concierge, security, admin.
- Жильцы видят свои заявки, персонал/админ — все.
- Управление заявками (одобрение/отклонение) доступно staff и admin.

## Ключевые функции

1. Заявки
   - Пропуска: разовые, временные, постоянные.
   - Техзаявки: электрик/сантехник.
   - Статусы: pending, approved, rejected, accepted, arrived, scheduled, expired.
   - Планирование отправки на будущее.

2. Пост охраны
   - Отдельный режим с быстрыми действиями (пропустить/отказать/отметить вход).
   - Выделенные подпотоки: активные, временные, техслужба.
   - Проверка по чёрному списку.

3. QR-сценарии
   - Генерация QR для пропусков.
   - Сканирование и обработка в отдельных модалках.

4. Чат
   - Групповой чат с реакциями, reply, редактированием и удалением сообщений.
   - Фото, предпросмотр, маркеры доставки/прочтения.

5. Админка
   - Статистика, пользователи, заявки, перм-списки, журнал, чёрный список.

## Данные и хранение

- Инициализирующие демо-данные лежат в слайсах (`requests/users/perms/blacklist`).
- Состояние сериализуется в localStorage с деградацией при quota exceeded (очистка старых фото).
- Фото заявок/аватарки сохраняются как data URL (в demo).

## Firebase-готовность

- В `src/config/features.js` флаг `FIREBASE_LIVE`.
- `src/services/firebaseService.js` инкапсулирует все live API; пока возвращает заглушки в demo.
- Для выхода в production нужно подключить реальные реализации и auth.

## UX и platform-фичи

- Тёмная/светлая/auto тема, сохранение выбора.
- Push уведомления и app badge (где поддерживается).
- Service Worker регистрируется из `utils.js`.
- Мобильные паттерны: swipe/long-press в чате, mobile nav.

## Текущее состояние качества

- `npm run build` успешен.
- Есть ESLint warnings:
  - mixed operators в `ReqCard.jsx`.
  - unused var `approveRequest` в `GuardPostMode.jsx`.

## Список проблем и рисков

### 1) Безопасность и прод-готовность
- Логин сейчас демо-формата (любой OTP), нет реальной проверки пользователя/сессии.
- Нет backend-валидации критичных действий (статусы заявок, создание пользователей, изменения ролей).
- Нет явного audit trail на уровне сервера (кто/когда/что поменял) — только клиентская история.

### 2) Архитектурные риски
- В части экранов бизнес-логика слишком плотно связана с UI-компонентами (сложнее тестировать и переиспользовать).
- Глобальный стор монолитный: по мере роста фич будет расти стоимость изменений и риск регрессий.
- Demo/live ветвление местами размазано по UI + сервисам, что усложняет поддержку двух режимов.

### 3) Надёжность данных
- localStorage используется как основной persistence в demo; при больших фото высок риск quota issues.
- Сложная сериализация/десериализация дат и media-ключей повышает риск edge-case ошибок.
- Нет схемной валидации объектов заявки/чата/пользователя перед записью в стор.

### 4) UX и эксплуатация
- Есть накопленные lint warning’и — признак мест, где код уже стал хрупким.
- В реальном сценарии нужен более предсказуемый контроль push-уведомлений, дублей и шумов.
- Нужны явные loading/error/empty-state стандарты для всех ключевых экранов.

## Что бы я рефакторил в первую очередь

1. **Auth + RBAC на сервере**  
   Вынести проверку ролей/прав в backend rules, добавить реальную авторизацию (Firebase Auth/IdP) и проверку токенов.

2. **Domain/use-case слой поверх UI**  
   Перенести сложные сценарии заявок/охраны/чата из компонентов в хелперы/use-cases (чистые функции + unit tests).

3. **Разделить store по bounded context**  
   Явно отделить `requests`, `chat`, `users`, `security` контексты; снизить связанность и упростить расширение.

4. **Валидация входных данных**  
   Добавить runtime-схемы (zod/yup) на вход формы и в reducer actions.

5. **Медиа-стратегия**  
   Убрать хранение крупных dataURL в localStorage (перейти на Storage + ссылочные метаданные).

6. **Набор базовых тестов**  
   Начать с критичного: права доступа, переходы статусов заявок, истечение временных пропусков, blacklist matching.

## Что улучшить в первую очередь

1. Включить реальную аутентификацию (Firebase Auth или альтернатива).
2. Закрыть ESLint warnings.
3. Перенести крупные доменные операции из UI-компонентов в hooks/services.
4. Добавить тесты критических правил доступа и жизненного цикла заявок.
5. Подготовить backend-валидацию и аудит-лог для production.
